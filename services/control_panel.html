Quick-start guide

      
New here? Follow these five steps to go from a fresh container to a fully-tested dispatcher filters. Already know what you're doing? Jump straight to any tab above — this guide is here when you need it.

    


    

      

        
1

        

          
Choose a module version

          
What: The AEM Dispatcher module (mod_dispatcher.so) is the Apache plugin that evaluates your filter rules and decides what gets to fetch files from AEM (HTTP 2xx) or not (HTTP 404). Nothing proprietary is baked into the image — the module is downloaded fresh from Adobe's CDN each time you switch.

          
How: In Server Controls → Dispatcher module version, pick the version that matches your production AEM environment, then click Download & switch module. Apache will restart automatically. Not sure which version? Use the latest, or check your team's AEM release notes.

          
        

      

    


    

      

        
2

        

          
Verify container health

          
What: Before testing filters the two Apache processes must be running. The dispatcher (port 80) is the real dispatcher handling requests. The renderer (port 4503) is a lightweight mock AEM publish instance that always returns HTTP 200 — it gives the dispatcher something to proxy to so filter decisions are realistic.

          
How: Check the status chips in Server Controls → Status. Both should show running. If they are not, click Restart dispatcher. Status auto-refreshes every 5 seconds.

          
        

      

    


    

      

        
3

        

          
Add or generate your filters

          
What: Dispatcher filter rules control which HTTP requests are allowed to reach AEM publish and which are rejected (returning 404). Rules live in _filters.any files and are evaluated top-to-bottom — the last matching rule wins.

          
How: Go to the Filters tab:

          
    
            
  • Pick a Default policy first. Deny all is the realistic production setting — rejects everything unless you explicitly allow it. Allow all is useful for testing deny rules in isolation.
    • 
            
  • Paste rules directly into the editor — use this if you already have a _filters.any file from your production environment (typically found in /etc/httpd/conf.dispatcher.d/filters/).
    • 
            
  • Generate rules from requests — paste raw HTTP lines (e.g. from access logs) and rules are created automatically for you.
    • 
          

          
Click Save & Restart after editing. Your filter file is volume-mounted to the host so changes persist across container restarts.

          

            
⚠ Security insights

            
    
              
  • Using fewer than 2 matchers per filter is a strong sign of a poorly written, insecure rule. Bypassing one matcher is far easier than bypassing many — the more specific matchers you combine, the harder the rule is to circumvent.
    • 
              
  • Dispatcher filters won't stop Apache httpd from serving files directly from the document root or cache. A rejected request never reaching AEM doesn't mean a previously cached file won't be served. Always clear the cache when testing filter changes outside of the sandbox.
    • 
              
  • When the dispatcher rejects a request it responds with a 404 Not Found, not a 403 Forbidden — it signals the resource doesn't exist rather than that access is denied.
    • 
              
  • Dispatcher filters are not a substitute for a Web Application Firewall. They define which content to expose, not how to fight malicious traffic. Use ModSecurity or a cloud WAF (AWS WAF, Cloudflare, etc.) alongside your dispatcher rules for real threat protection.
    • 
            

          

          
        

      

    


    

      

        
4

        

          
Test your filters and share the results

          
What: Each path in Good paths is expected to return HTTP 200 (the dispatcher should allow it). Each path in Bad paths is expected to return HTTP 404 (the dispatcher should reject it). Tests hit the live running dispatcher — no mocking.

          
How: Go to the Testing tab. Click Generate tests from filters to auto-populate both lists from your current rules, or paste in real paths from your site. Click Run tests and review the colour-coded results table. Use Export JSON or Export CSV to share results with your team or save a baseline for future comparisons.

          
        

      

    


    

      

        
✓

        

          
WIN!

          
When you test in the sandbox you can verify your rules are functional and play nice by themselves and with other rules without needing to affect an actual AEM hosted system.  It's all on your workstation locally.  Makig sure your rules are working as expected before deploying to production. Then your dispatcher rules are verified and your results are documented. The same 002_web_filters.any file from the filters/ volume mount is ready to deploy to your production AEM dispatcher — tested and proven.

        

      

    


    

      
Bonus: reading the logs

      
The Logs tab links to live-streaming views for all processes. Here is what each log tells you:

      
    
        
  • dispatcher.log — The most useful log. With trace logging enabled you will see every filter rule evaluated per request. Look for Filter lines to trace exactly why a request was allowed or rejected.
    • 
        
  • filter-test.log — Output from the filter parser script. Check this if rules are not appearing in the Active filter rules view.
    • 
        
  • error_log — Apache / dispatcher startup and runtime errors. First stop when the dispatcher fails to start.
    • 
        
  • access_log — Every HTTP request that reached Apache. Useful for confirming your test client is actually hitting the dispatcher.
    • 
      

      
Tip: run this in your terminal for a combined real-time stream of all container output:

      docker compose logs -f
